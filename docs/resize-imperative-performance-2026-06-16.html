How it works — at a glance 🎯

Resizing ONE object used to re-render the whole canvas's React tree

Drag was already smooth — it commits to Redux zero times mid-gesture. Resize kept dispatching every frame, and every dispatch re-ran the giant MainCanvas component.

❌ Before (per resize frame)

onResize → dispatch(redux)
└─ MainCanvas re-render (3900-line reconcile)
…a commit (+ undo churn) every frame

✅ After (per resize frame)

onResize → store.setLiveResize(id)
└─ only THAT object re-renders
…MainCanvas untouched · commit once on release
Why a store, not imperative DOM?

We want true reflow (text re-wraps, image re-crops, calendar grid re-lays-out) — not a uniform scale(). Letting React render the object with the live geometry gives correct reflow for free, for every type, with no per-type DOM surgery.

1. The diagnosis (measured, not guessed)

The lag was localized by elimination — each observation ruled a hypothesis out:

Type-independent (text/shape/image equal)Image-specific raster

Disabling Moveable snapping changed nothingMoveable re-measure / snap

What remains is a fixed per-dispatch cost invisible to paint/highlight = the synchronous MainCanvas reconciliation forced by flushSync on every per-frame dispatch. Confirmed: drag (0 dispatches) is smooth; resize (per-frame dispatch) is not.

2. The mechanism

2.1 Per-object override store

A module-level Map<id, override> + useSyncExternalStore. setLiveResize(id, {width,height,image?}) notifies only that id's subscribers; getLiveResize returns a cached ref (or null) so there's no idle re-render and no getSnapshot thrash.

2.2 Renderers merge the override

const liveResize = useLiveResize(rawItem.id);
const item = liveResize ? { ...rawItem, ...liveResize } : rawItem;

null at all other times → a complete no-op outside an active resize. During a resize, only the one object's component re-renders (canvas + footer copy) and React reflows the real geometry.

2.3 ItemDragger drives it imperatively

  • onResizeStart — seed resizeWorkingRef from the live object (the image-cover math chains off it, reproducing the old flushSync chain since Redux is frozen during the gesture).
  • onResize — compute next geometry (image cover / text measure+clamp / plain size) and setLiveResize(id, …). No Redux dispatch.
  • onRender — wrapper translate via cssText (canvas) + mirrored to footer (applyLiveObjectTransform); skips the per-frame dispatch for imperative types.
  • onResizeEnd — one history:true commit of width/height/image (+ x/y if a top/left handle moved the wrapper), then clear the override. Committed value == last override → no jump.

2.4 Text's self-managing effects are guarded

Canvas Text has a font auto-grow effect and a Firefox height-fix effect that dispatch height and are keyed on item.width/height. They early-return while isLiveResizing and re-run once the resize commits ("can't shrink below content" is also kept live via a measure+clamp in onResize).

5. Safety audit (verified by reading the code)

  • Undo/redo — zero history during the gesture; exactly one history:true commit on release. One Ctrl+Z restores fully.
  • Image crop/cover — math ported verbatim, chained through the working ref → same result as the old flushSync chain (no drift).
  • Rotated objects — rotation is not taken from the gesture (which can report 0 during resize); the pre-resize angle is preserved on commit and in the footer mirror.
  • Text "can't shrink below content" — preserved (live measure+clamp; guarded effect re-validates on commit).
  • Save / export / order / themes — all read Redux (committed value); no blob:/URL handling, so nothing can leak.
  • Locked / customer-disabled — resize gated by isAdjustable; untouched.
  • Idle cost — useLiveResize returns null when not resizing → no extra re-renders, ever.
  • Calendar / QR / Shape / Sticker — pure renderers (no dispatch), so wiring cannot re-introduce the lag or fight the override.

6. Gotchas (do not regress)

  • Never coalesce the resize dispatch onto rAF — it desyncs from Moveable's flushSync selection box. The fix is no per-frame dispatch, not fewer.
  • Text's dispatching effects MUST stay guarded (isLiveResizing) — without it they dispatch height per frame, re-introducing the lag.
  • MainCanvas must NOT subscribe to the store — the store is consumed only by leaf renderers.
  • Commit on release must equal the last override (read from resizeWorkingRef) — else the object jumps on release.
  • Calendar width/height are default-from-item params; re-derive them from the merged item or the override won't take effect.

7. Verification checklist

  • "Highlight updates": resizing any object highlights only that object.
  • Profiler: MainCanvas shows 0 commits during a resize, 1 on release.
  • Smooth at 4× CPU throttle for image, text, shape, sticker, QR, calendar.
  • Footer thumbnail tracks live (size + position) for every type.
  • Top-left-corner resize and rotated-object resize both track on canvas + footer.
  • Ctrl+Z after a resize = one clean undo, every type.
  • Text still cannot shrink below content; auto-grow works after release.
  • Save/export reflect the final committed size.
