How it works — at a glance 🎯

① The lag: moving ONE object repainted ALL of them

Imagine a classroom where the teacher asks one kid a question, but the whole class has to stand up and sit down. That's what was happening: drag one photo, and all ~30 objects on the page re-rendered 60 times a second.

❌ Before — everyone re-renders

🔴 every object repaints — slow

✅ After — only the one you touched

🟢 only the dragged one repaints — fast

The trick: give every object a stable "name tag" (a WeakMap) so React can tell "this one didn't change, skip it."

② How a drag works now: move first, save once

Before, every wiggle of the mouse was written to the "save file" (Redux) — and writing the save file is what woke up the whole heavy canvas. Now we move the picture directly on screen and only write the save file once, when you let go.

③ But the footer is a "mirror" — keep it in sync

The little page thumbnails at the bottom show the same objects. Since we stopped writing the save file mid-drag, the footer wouldn't know to move. So during a drag we also nudge the footer's copy directly — both move together, still no save-file writes.

Same idea for the blue selection box: a tiny loop keeps re-measuring it each frame so it follows the photo, then snaps back when you let go.

1. The problem

At 4× CPU throttle the main canvas stuttered during interaction while the sidebar/footer did not. The cost was re-render fan-out and per-frame work, not raster:

  • One interaction frame → one redux dispatch → MainCanvas (~3900 lines) re-renders → its N-object .map runs → every object re-renders (memo defeated).
  • Inner-image zoom changed <img> width/height → browser re-rasterized the full-res bitmap every frame.
  • MainCanvas re-rendered ~60×/sec during any gesture.

2. Root causes & fixes

2.1 Re-render fan-out → stable identities + memoization

getAllObjectsSortedByZIndex spread-copied every object ({...obj}) each recompute → every item was a new reference → React.memo failed (and Text wasn't memoized) → all N objects re-rendered. Now the selector wraps each object through a WeakMap keyed by the underlying object; Immer keeps unchanged objects' identity, so they reuse the same wrapped reference → memo holds → only the changed object re-renders.

BEFORE (per drag frame)                AFTER (per drag frame)
 dispatch                               dispatch
   └─ getAllObjects (new array,           └─ getAllObjects (WeakMap reuse:
       {...obj} → ALL new refs)               only changed obj is a new ref)
        └─ MainCanvas re-render                └─ MainCanvas re-render
             └─ Photo×30 re-render  ✗               └─ Photo×1 re-render  ✓
                Text×N re-render  ✗                    (others memo-bail)

2.2 Full-res raster on zoom → GPU transform

The canvas sized the image with width = image.width * scale; changing that re-rasters the bitmap every frame. Now the image renders at fixed intrinsic size and zoom is transform: translate(pos) scale(scale) (transformOrigin: 0 0) → decoded once, GPU samples the texture. Clip/mask/flip are frame-space and unaffected.

2.3 Per-frame redux dispatch → imperative gestures

react-moveable already moves the element imperatively (cssText), and the image uses a GPU transform — so the per-frame dispatch is redundant for the visual; its only job is persistence. Drag/rotate/zoom/pan dispatch nothing per frame and commit once on gesture end. MainCanvas commits during a gesture drop from ~60/sec → ~0. (Resize keeps dispatching — its size visual comes from redux width/height.)

2.4 Imperative gestures broke the footer → mirror to footer DOM

The footer reads positions/sizes from redux, so imperative gestures froze it. Helpers in progressiveImage.js mirror the gesture to the off-canvas DOM copies (footer renders the same objects with the same data-id/data-id-t):

  • applyLiveImageTransform(id, image, canvasRootEl) — inner zoom/pan (canvas <img> transform; footer <img> margin + scaled size).
  • applyLiveObjectTransform(id, x, y, rotation, canvasRootEl) — object drag/rotate (both <g class="page-item" data-id-t>).

canvasRootEl (main <svg>) is passed so the canvas copy is skipped. No redux → no MainCanvas re-render → footer stays live.

2.5 Selection box tracking during imperative pan

The moveable box re-measured when activeObjectprops changed (every frame via the old dispatch); imperative pan stopped that → the box froze. Fix: a rAF loop calls updateRect() every frame while isDragger is true, and on pan end (incl. a bare pan-handle click that briefly shows the box-expanding GhostImage) it re-measures so the box snaps back instead of staying oversized.

2.6 Zoom-after-undo jump

The undo wrapper preserves activeObjectprops across Ctrl+Z, so it kept the last-zoomed scale even after the object reverted. The zoom base now reads from the live object (activeObjImageRef, synced from allObjects + activeObject) → re-zoom starts from the reverted scale.

2.7 Per-frame DOM query during inner pan/zoom → cache the targets

The footer mirror (§2.4) finds an image's on-screen <img> elements with document.querySelectorAll(...). The first version called that every frame of a pan/zoom — pan twice (images + GhostImage), zoom once — and the scan walks the whole DOM, including every footer thumbnail. That per-frame scan was a residual "little lag."

The gesture is imperative (no re-render), so those elements don't change mid-gesture — no reason to re-find them. The helper is split into query-once + apply-to-cached: queryLiveImageEls(id) runs the scan once; applyLiveImageTransformEls(els, image, canvasRootEl) applies to the cached elements; applyLiveImageTransform(id, …) stays as a convenience for non-hot callers. Pan (Photo.jsx) caches { imgEls, ghostEl } in panElsRef on the first frame and clears on mouse-up; zoom (Canvas.jsx) caches zoomImgElsRef per wheel session and clears on commit. Net: N DOM scans per gesture → 1. Pure DOM-lookup change — same elements, same writes, so redux/save/theme/export are untouched.

6. Gotchas (do not regress)

  • The footer reads from redux — any gesture made imperative MUST be mirrored to the footer DOM, or it freezes.
  • Undo preserves activeObjectprops — never use it as a gesture base; read the live object.
  • No mid-zoom history:true snapshot — it re-renders and resets the imperative transform (the "glitter"). Commit on pause.
  • Selection box needs the rAF updateRect() loop during pan + a re-measure on pan end.
  • Full-cover spread spine offset isn't re-applied in the footer mirror during a live drag (catches up on commit).
  • Cache the gesture's DOM targets, don't re-query per frame (panElsRef/zoomImgElsRef) — and clear them on gesture end (stale nodes would point at a removed object). Use applyLiveImageTransformEls on the hot path.
  • Deferred (still per-frame dispatch): multi-select group drag, object resize, whole-canvas zoom.

7. Verification checklist

  • DevTools "Highlight updates": dragging highlights only that object.
  • Profiler: MainCanvas = 0 commits during a drag, 1 on release.
  • Zoom/pan/drag/rotate smooth at 4× CPU throttle.
  • Footer tracks live for drag, rotate, resize, image zoom, image pan.
  • Selection box follows the photo during pan; snaps back on a pan-handle click.
  • Zoom → Ctrl+Z → zoom again starts from the reverted scale.
  • Inner pan/zoom stays smooth with many footer thumbnails present (no per-frame DOM scan).
  • Export / save reflect the final committed position.
