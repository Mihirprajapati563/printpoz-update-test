Firefox + react-moveable zoom fix Golden Path

Session notes — 2026-06-24 · the selection box detached from objects in Firefox when the canvas was zoomed. Root cause: CSS zoom. Fix: user-zoom as transform: scale, Moveable kept inside the transform, control handles counter-scaled with an inverse zoom prop.

Why CSS zoom was the root cause 🎯

CSS zoom is non-standard and only recently standardized by the W3C. The standardization changed how zoom maps into getBoundingClientRect() — from unscaled to zoom-scaled rects. Chromium shipped this on-by-default in 128; Firefox in 126. react-moveable builds its control-box matrix from getBoundingClientRect and is designed around CSS transform; its type defs say "Because getBoundingClientRect is used, css zoom … cannot be used." With CSS zoom on the wrapper, Firefox (new behavior) and the running Chrome (older behavior) disagreed → the box drifted by the scale factor.

Not "Firefox-only." It's a standardized-zoom issue — modern Chrome (128+) hits it too; a "working Chrome" was simply older. The fix is to stop using CSS zoom, not to special-case Firefox.

The Golden Path — DOM structure

Moveable lives INSIDE the transform; the sizer carries the scaled footprint

The single most important rule: <Moveable> renders inside the transform: scale box (shares the target's transform context). Putting it outside changes the target's offsetParent and detaches the box at every zoom in every browser.

.pages-outer  (overflow:auto when zoomed)
└── .canvas-zoom-sizer     width/height = base × scale + HANDLE_BUFFER×2  (MEASURED)
    │                      → scaled layout footprint → .pages-outer overflows
    │                        → pan works + room for the selection box/handles
    └── .canvas-box         transform: scale(canvasBoxScale); transformOrigin: 0 0;
        │                   position:absolute; top/left: HANDLE_BUFFER; willChange:transform
        ├── .ruler-reserve-wrap   transform: scale(0.95)   (ruler gutter; unrelated)
        │   └── <svg> … objects (Moveable targets)
        ├── <ItemDragger/>   ← INSIDE .canvas-box → shares the target's transform
        └── <SafeAreaDragger/>     context. zoom={1/canvasScale} keeps lines 1px.

The mechanism (3 files)

1. Canvas.jsx — transform + sizer + measured footprint

const canvasBoxScale = canvasScale * mobileCoverMult;   // was the CSS zoom value

<div className="canvas-zoom-sizer"
     style={{ margin:"auto", position:"relative",
              width: zoomFootprint?.w, height: zoomFootprint?.h }}>
  <div ref={canvasBoxRef} className="canvas-box"
       style={{ position:"absolute", top:HANDLE_BUFFER, left:HANDLE_BUFFER,
                transform:`scale(${canvasBoxScale})`, transformOrigin:"0 0",
                willChange:"transform" }}>
    …pages… <ItemDragger/> <SafeAreaDragger/>
  </div>
</div>
  • Footprint is measured: a useLayoutEffect reads canvasBoxRef.offsetWidth/Height (pre-transform → stable base) and sets zoomFootprint = base × canvasBoxScale + HANDLE_BUFFER×2. Use offsetWidth (≈ max-content), not a fit-content wrapper — fit-content clamps to the sizer and feedback-collapses the canvas when zoomed out.
  • HANDLE_BUFFER (44px) on every side gives the box's borders + handles room so .pages-outer { overflow:hidden } can't clip them.
  • The two overflow-detection effects now read canvasBoxScaleRef.current instead of getComputedStyle(canvasBox).zoom (now 1).

2. ItemDragger.jsx — inverse zoom + snap digits off

const moveableZoom = 1 / (canvasScale || 1);   // INVERSE → control lines stay 1px on screen
// on BOTH the single-object and group Moveable:
<Moveable zoom={moveableZoom} isDisplaySnapDigit={false} isDisplayInnerSnapDigit={false} … />
  • zoom is the inverse scale. At 50% (scale 0.5) → zoom 2 → lines render 2px local → 1px on screen. (zoom={canvasScale} was the bug — it made lines thinner, which Firefox rounds to 0.)
  • Snap-distance digit text hidden — Moveable sizes it as fontSize × zoom, huge under the inverse zoom. Snapping/guidelines still work.

3. SafeAreaDragger.jsx

Same inverse zoom={1/canvasScale} on its Moveable (also inside the scaled .canvas-box).

Verification (runtime — both Firefox + Chrome)

Must pass

Known caveat (pre-existing)

Marquee/paste calc (toCanvasCoords → /zoomRatio) doesn't account for canvasBoxScale, so rubber-band select / paste-at-cursor can be slightly off while zoomed. Same root cause; fix the divisor if it surfaces.
