Built 2026-06-25. Desktop renders export pages LOCALLY instead of POSTing SVG to the server's `exportAsJpeg`. This is REQUIRED, not just faster: user photos are local `app-assets://` files the server can't see, so a server render would be blank. Same applies to 3D-preview textures. See [[local-image-store]].

**⚠⚠ 2026-07-08 — TILING IS DEAD, replaced by CDP `Page.captureScreenshot`. Everything below about tiles/capturePage/paint-events/settle is SUPERSEDED — kept only as history.** The tiling approach (capturePage per screen-bounded tile + stitch) was fundamentally broken: capturePage is monitor-clamped, so tiles captured STALE frames under GPU contention → scrambled/duplicated/cut exports, and the timing budgets made a 6000×9000 page take ~90s (measured `paintHits ~1/40` — the paint-event fast path never fired on the user's machine; blank-retry punished white margin tiles). **NEW approach (`exporter.ts` fully rewritten):** render the WHOLE page in ONE shot via the Chrome DevTools Protocol. `webContents.debugger.sendCommand('Emulation.setDeviceMetricsOverride',{width:W,height:H,deviceScaleFactor:1})` forces the layout viewport to full page size DECOUPLED from the monitor clamp (this is the whole trick), then `Page.captureScreenshot({clip:{x:0,y:0,width:W,height:H,scale:1},captureBeyondViewport:true,fromSurface:true,optimizeForSpeed:true,format,quality})` grabs the full page → base64 → Buffer. **Empirically validated (standalone Electron harness): 6000×9000 correct in ~1.5–2s (vs ~90s), exact dims, foreignObject text + rects all render.** Hard limit 16384px/axis (compositor texture); guard at `MAX_AXIS=16000`, else band-tile the oversized axis and stitch in the renderer. Result shape now `{width,height,format,bytes?,tiles?}` — `bytes` = the whole page as one image (common), `tiles` = bands only past the cap. `localExport.js` wraps `result.bytes` in a Blob DIRECTLY (no canvas stitch/re-encode) + embedImageDpi.
**CRITICAL window model (learned the hard way in the harness):** (1) `Page.captureScreenshot(fromSurface)` HANGS forever on a `show:false` window that never allocated a compositor surface → the window MUST be `showInactive()` off-screen at `-32000,-32000` (composites, never visible, no focus steal). (2) Creating a NEW window per page makes page 2+ fail loading with `ERR_FAILED` → use ONE PERSISTENT window, load a blank shell ONCE, and swap each page's `@font-face` CSS (`#pp-fontfaces`) + SVG (`#host.innerHTML`) via the DOM — NO per-page navigation. Window reused across the export, destroyed after `IDLE_DESTROY_MS=60s`. (3) `offscreen:false` (NOT true — OSR reintroduces the clamp). (4) DSF pinned to 1 + verify output size (HiDPI-doubling is a version-dependent Chromium regression). (5) white `setDefaultBackgroundColorOverride` (JPEG transparent→black). FIFO queue still serializes (one window, one render at a time). Test harness lived at repo-root `_cdp_test*.js` (deleted after validating).

**How (historical, tiling):** an offscreen `BrowserWindow{offscreen:true}` = local headless Chromium → renders SVG `<foreignObject>` text + filters + `app-assets://` images. One window at a time (FIFO queue).

**⚠ capturePage is HARD-CLAMPED to screen resolution** (proven: a 5400×3600 window captured only top-left ~1920×1040; the OSR `paint` event is clamped too; a 5400 *window* even fails with GPU `ERR_FAILED`). So a print-size page CANNOT be captured in one shot. **Solution = TILING:** load the full-size SVG once in a screen-bounded window inside `#wrap`, translate `#wrap` per tile via CSS transform, `capturePage()` each tile (1:1), return tiles to the renderer, which stitches them onto a full-size `<canvas>` (NOT screen-limited; tiles are raster so `toBlob` isn't foreignObject-tainted). `exporter.ts` returns `RenderSvgResult{width,height,tiles[]}`; `localExport.js renderSvgToBlobLocal` stitches. A large *window* fails but a small window with a large translated child is fine. Also MUST wait for `<image>` decode before capture (`WAIT_IMAGES_JS`) or photos capture gray. Validated end-to-end against real assets 2026-06-25.

**Files:** `electron/main/services/exporter.ts` (`renderSvgToImage`), `electron/main/ipc/export.ipc.ts` (`export:renderSvg`, dim cap 20000), contract/preload `desktop.export.renderSvg`. Renderer: `src/library/utils/services/export/localExport.js` (`renderSvgToBlobLocal` + `buildExportFontCss`), font registry added to `fontLoader.js` (`getCustomFontSources`/`getGoogleFontSources`). Wired: all 7 `exportAsJPG` sites in `useExportPages.js` (via one `exportPageBlob()` helper that branches on `localExportEnabled`) + `getTrimmedPreviewTexture.js`. `services/export/index.js` `SvgToJPG` is DEAD code (broken imports, unused) — left alone.

**Fonts (offscreen doc is a fresh document with NO loaded fonts):** custom fonts embedded self-contained as base64 `data:` URIs (offline); Google fonts via inlined css2 stylesheet (ONLINE — per offline-assets policy, Google fonts aren't cached). `fontLoader.js` records every loaded font's source so export can rebuild @font-face.

**DPI/quality (2026-06-25 update):** `page.width/height` = `canvasSize.width/height` (the WORKING canvas px) — NOT extra-DPI-scaled. So default export was soft/low-res (a 6000px photo downsampled into a 2400px page → ~1MB). Fix: `useExportPages` computes `exportScale = max(1, canvasSize.dpi/200)` and passes it to `renderSvgToBlobLocal`, which renders the page at `w*scale × h*scale` (SVG viewBox keeps layout; photos sample higher-res). Tiles encoded JPEG 98, final stitch JPEG 0.95 (avoids the old lossy 92→92 double-encode). `exporter.ts` logs `[exporter] render page WxH tiles CxR svgLen=N` per page.

**Tiling = SVG viewBox (FINAL approach, 2026-06-25):** each tile is rendered NATIVELY by setting the page SVG's own `viewBox="tx ty TILE_W TILE_H"` + width/height=TILE (via `applyTileBox` executeJavaScript on `#host > svg`), NOT by offsetting a giant 4800px layer (left/top or CSS transform — those mis-composited → scrambled tiles). viewBox renders ONLY the visible region (less work, no off-screen layer) and is pixel-exact at boundaries. PROVEN end-to-end: stitched a 4800×2400 page where a corner-to-corner diagonal line + a horizontal boundary line + an image straddling tile seams were all perfectly continuous (no jog/gap), under GPU contention. Combined with paint-frame capture below.

**Export QUALITY = lossless tile pipeline (2026-06-25):** the client looked SOFTER than the server at the SAME resolution because the tiled pipeline encoded each tile JPEG (lossy) THEN re-encoded the stitched canvas JPEG (lossy) — a double-compression the server's single-pass `nodeHtmlToImage` (quality:100) never does. FIX: `exporter.ts` encodes tiles **PNG (lossless)** always; `localExport.js` stitch sets `ctx.imageSmoothingEnabled=false` and does ONE final encode (JPEG 0.96 / PNG). Now the only lossy step is the final encode = matches server quality. NO supersampling — user wants SAME pixels (canvasSize.dpi=200 → scale 1); higher quality came from removing the double-encode + forcing the large variant, NOT from more pixels. (PNG tiles = bigger IPC payload, transient.)

**Large variant at export:** `renderPhotoObject` in `generatePageSvg.js` now prefers `item.urls.find(size==='large').url` (falls back to `item.url`) — the canvas may DISPLAY a lighter progressive variant but the export must embed the full-res large image.

**Capture sync — THE fix for scrambled/misaligned tiles (2026-06-25):** the real export produced SCRAMBLED pages (text/photos duplicated and shifted — tile (1840,0) showed tile (0,0)'s content) because `capturePage()` returns the PREVIOUSLY-composited frame, so it captured the STALE offset under GPU contention with the live editor window. FIX: capture the **paint event's OWN frame** — `webContents.on('paint', (_e,_d,image)=>...)` feeds `nextPaint()` the NativeImage; per tile: set offset → `invalidate()` → discard one (possibly in-flight) paint → use the NEXT paint's image (fallback `capturePage`). Proven with per-tile markers (R0C1 in the col-1 tile, R2C2 in the bottom-right) under a competing GPU-busy window — perfect alignment. Do NOT revert to capturePage-after-delay. `isLikelyBlank` retry kept as a secondary net. Note: the scrambled exports were also tiny (KB) because the misaligned/flat regions compress to nothing — correct full-photo tiles produce proper multi-MB files.

**DPI reality:** real export logs showed pages at base size (spreads 4800×2400, special pages 2400×2400) → `canvasSize.dpi=200` so `exportScale=1` (no upscale). Those base sizes are already print-res; the quality win is the JPEG 98-tile/0.95-final encode (was lossy 92→92). If higher-res is wanted, switch to a fixed multiplier.

**⚠ Special/cover pages export only partially (render-engine RULED OUT):** proven the tiling engine captures heavy multi-row foreignObject pages fully (validated against real assets). So the partial last/second-last (photobook special/cover) pages are a DATA problem — the generated `svgData.svgContent[]` for those pages (produced at canvas-view time by `SVGRenderer` → `generatePageSvg`; special pages use `isSingleSideExport`→halfWidth) has wrong dims/viewBox or is stale. Diagnose via the `[exporter]` per-page log (compare special-page WxH vs others). Photobook special pages = cover, pageIndex 0, 1, and totalPages-2 (`generatePageSvg.js:807`).

**Font disk cache (2026-06-25):** `fonts:cacheGet/cachePut` IPC + `font-cache.service.ts` store downloaded WOFF2/TTF under `userData/cache/fonts/<sha256-of-url>.<ext>`. `fontLoader.loadFontFile` (desktop) checks the cache first, downloads + caches on miss, builds `FontFace` from the ArrayBuffer (web keeps url() source). Fonts immutable per URL → cached indefinitely.

**⚠ The export window MUST use `sandbox: false`** (`exporter.ts` webPreferences). A sandboxed renderer CANNOT load a `data:` URL — the export HTML shell fails every render with `ERR_FAILED`, which the export flow swallows → "export shows loading then stops, no file, no error". Proven empirically: sandbox:true → THREW; sandbox:false → works. The window loads only our own SVG, has no preload, and denies navigation/popups, so the sandbox bought nothing. (Dev runs in electron:dev → must RESTART electron, not just HMR the renderer, to pick up a new main bundle.)

**Native save + progress UX (2026-07-07):** file SAVE now goes through native OS dialogs on desktop (web keeps `saveAs`/`<a download>`), fixing the "asks for the folder again and again" bug (multi-file JPEG/PNG export used to fire one `<a download>` per page = N save dialogs). New module `src/library/utils/services/export/saveHelpers.js`: `saveSingleFile({blob,filename})` → one native Save-As (PDF/ZIP/single image), `saveMultipleFiles({files,title})` → ONE folder picker (`desktop.dialog.openFolder`) then writes every file into it. Both return `{saved,canceled}` so the UI treats a cancelled dialog as not-success. New IPC `dialog:openFolder` (`showOpenDialog` openDirectory+createDirectory) — contract in `electron/shared/ipc.ts`, handler in `dialog.ipc.ts`, preload `desktop.dialog.openFolder`. All ~8 save sites in `useExportPages.js` route through the helpers. Centered `<ProgressModal>` (theme-var styled, format-agnostic wording, page X/Y + %) now renders for the EDITOR export too (`Canvas.jsx`, gated `!isPreviewBook` — preview flows already render their own); `exportPageSVG` drives it in every branch (render 0-80%, assemble 80-92%, save 92-100%). The Canvas "Preparing your pages…" capture overlay was re-themed to match. **Export "sometimes doesn't start" = WEDGED FLAGS (fixed):** `exportPage()` only sets `initilized(true)`; if capture threw, `setStartExport(true)` never ran and nothing cleared `initilized`/`exportInProgressRef`, so the next click's `setInitilized(true)` was a no-op (no false→true transition) → dead. Fix: try/catch in `triggerDomCaptureAndExport` + the single-page else-branch + the empty-svgContent early return all clear `initilized`/`startExport`/`isCapturingPages`/ref (and toast) so a retry always gets a clean transition. Do NOT "reset-then-set" in one React batch (collapses to no transition).

**Still caveats:** (1) Google-font export needs internet. (2) Not yet runtime-verified in the GUI — needs a real export test (PDF + ZIP + single JPEG + 3D preview texture all show the local photos, and the native save/folder dialogs).
